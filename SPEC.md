# bgh-proxy — design & architecture

A transparent GitHub API proxy that adds per-repo/per-org/per-resource read/write access policy, audit logging, and token custody — so the powerful GitHub token never leaves the proxy host. Works with the stock `gh` CLI and any GitHub API client.

This document describes how the proxy is built and the security properties it aims to provide. For configuration, the policy language, and operational guidance see [README.md](README.md).

## Problem

The `gh` CLI's token model is coarse: `gh auth login` mints a token with `repo` (full read/write to every private repo you can see), `read:org`, and `gist`; it never expires; and it lives on the client (keychain, or plaintext `~/.config/gh/hosts.yml`). There is no read-only scope for private code, no per-repo granularity, and no audit trail. A compromised laptop means full access to everything.

`bgh-proxy` keeps the real token on a server and gives each client a narrowly-scoped credential plus an audit log.

## Architecture

The proxy impersonates the GitHub API. Clients (`gh`, `curl`, libraries) connect to it as if it were GitHub Enterprise. The proxy classifies the request, evaluates policy, swaps in the real token, forwards to `api.github.com`, and streams the response back.

```
┌────────────┐  /api/v3/repos/o/r/pulls   ┌─────────── bgh-proxy ───────────┐   real token   ┌────────────┐
│ gh / curl  │ ─────────────────────────▶ │ auth → classify → policy → fwd  │ ─────────────▶ │ GitHub API │
│ any client │  Authorization: <proxy tok>│        │           │            │                │            │
└────────────┘ ◀───────────────────────── │      audit       node          │ ◀───────────── └────────────┘
                  passthrough response     │      log         resolver      │   response
                                           └─────────────────────────────────┘
```

The real GitHub token never reaches the client.

## Request pipeline

`proxy.Handler.ServeHTTP` runs each request through:

1. **Authenticate.** GHE mode requires a proxy token in `Authorization`; it is looked up (constant-time) in the store. Socket mode ignores the client token and uses the single socket policy.
2. **GHE handshake shortcuts.** `GET /api/v3` returns a root document with `X-OAuth-Scopes`; `GET /api/v3/user` returns a synthetic identity. These let `gh auth login`/`status` complete without forwarding.
3. **Reject path traversal.** Any request whose path contains a `.`/`..` segment (after percent-decoding) is rejected `400`, so the path the classifier sees is the path GitHub would route.
4. **Classify** into one or more scopes (see below).
5. **Resolve node-ID scopes.** For a GraphQL mutation *or* a `node(id:)`/`nodes(ids:)` read, resolve referenced node IDs to repositories via GitHub (see below). Gated on the policy being able to act at that access level (write resolution skipped if the policy can never write; read resolution if it can never read). Write-path resolves bypass the node cache so a repo transfer cannot authorize a stale pre-transfer attribution.
6. **Evaluate policy** for every scope; deny (`403`) if any scope is denied.
7. **Forward.** Build a fresh upstream request: copy the client's headers (minus `Authorization`, `Host`, length, `Accept-Encoding`, `X-GitHub-Api-Version`, and hop-by-hop), set `Authorization: token <real-token>` and `X-GitHub-Api-Version`, default `Accept`/`User-Agent`/`Content-Type` only if absent. (`Accept-Encoding` is dropped so the upstream body arrives decompressed and the filters can parse it — see Security properties.) Stream the response back, but **not verbatim**: the custodian-revealing `X-OAuth-Scopes`/`X-Accepted-OAuth-Scopes`/`X-OAuth-Client-Id`, `X-GitHub-SSO` (SAML-SSO org reach), `Github-Authentication-Token-Expiration` (the custodian's expiry/lifecycle), `Set-Cookie`, and transport `Content-Encoding`/`Transfer-Encoding` headers are stripped, the request-side `Cookie` header is not forwarded upstream, and GraphQL/REST bodies are filtered/redacted (see Classification & Security properties). The GHE-handshake `GET /api/v3` returns a *fixed synthetic* `X-OAuth-Scopes: repo, read:org` (a constant to satisfy `gh`, independent of the custodian's real scopes), not the upstream value.
8. **Audit** the decision (allowed/denied + upstream status) as one JSONL line. The logger is async over a bounded (1024-entry) channel and is **best-effort**: under sustained overload it drops entries rather than block the request path, recording a `SYSTEM` entry with the dropped count. Treat the audit log as best-effort, not a guaranteed record of every decision.

## Classification

`classifier.Classify(method, path, body)` returns a primary scope plus any `Additional` scopes and, for mutations, the referenced node IDs. Access level: REST `GET`/`HEAD` = read, else write; GraphQL `query` = read, any `mutation` = write (fail-closed: unparseable/over-complex bodies are treated as write).

### REST

GitHub's REST API is path-regular:

```
/repos/{owner}/{repo}/{seg}/…  → repo = owner/repo, resource from {seg}
/orgs/{org}/…                  → org
/users/{user}/…                → org = user
everything else                → unscoped category from the first segment
```

The `resource` maps the first sub-segment (`pulls`, `issues`, `actions`, …) to a permission key. An unrecognized sub-segment yields a distinct `ResourceUnknown` sentinel: for a **write** under a rule that defines per-resource permissions, the policy fails closed rather than inheriting the base grant (so an unmapped write endpoint can't escape a per-resource `none`).

A **cross-fork compare** (`/repos/{o}/{r}/compare/{base}...{user}:{head}`, and the `dependency-graph/compare` variant) returns the *foreign* fork's commits/file-patches, which the path-repo scope doesn't cover and the REST response filter can't redact (the foreign content carries no per-element repository identity). The classifier parses the basehead and emits each foreign owner as an **additional repo scope** (the path repo name), so the policy must allow that fork too or the request is denied (round-16) — mirroring how the GraphQL filter handles `headRepository`/`forks` navigation. The same "a foreign repo named in the **request** must become a scope" rule applies to REST **bodies**: the migration (`POST /{orgs/{org},user}/migrations`) and CodeQL variant-analysis (`POST /repos/{o}/{r}/code-scanning/codeql/variant-analyses`) endpoints name their target repos in the JSON body, and the broad custodian token — not the client — has the reach to migrate/scan them, so the classifier parses the body `repositories[]`/`repository_owners[]` and scopes each (round-23). Without this a client with org-migration write but a per-repo `none` carve-out could have the custodian produce and hand back a full migration **archive** of a denied repo. (The variant-analysis `repository_lists` form names a saved list the proxy can't resolve offline, so its response `scanned_repositories`/`skipped_repositories` are additionally content-scrubbed.)

### GraphQL — multi-scope

A GraphQL document can touch many repositories/orgs in one operation, and GitHub executes all root fields. The classifier therefore walks the AST and collects **every** scope:

- `repository(owner:, name:)` → a repo scope (with resource inferred from its sub-selection);
- `organization`/`repositoryOwner(login:)`/`user(login:)` → an org scope (the login as the org), with cross-owner fork navigation still scanned. Because `user(login:"<custodian>")` and `repositoryOwner(login:"<custodian>")` resolve to the **authenticated viewer** when the login is the custodian's own, owner-private User fields selected through them (`email`/`gists`/`savedReplies`/`organizations`/`enterprises`/`projectsV2`/keys/…) are **un-floored to the same `gists`/`user_private` categories as the `viewer` root** (round-34) — so a token holding only base read for that login cannot read the custodian's private account data. The `organization` root is excluded (an org login can never equal a user login, so it never resolves to the viewer); `TestR34_UserRootPrivateFieldParity` derives the gated set from `viewerPrivateFieldCategory` so the two paths cannot drift. A `User` reached by **navigation** (an `author`/`uploadedBy`/`owner`/`...on User` edge) bypasses these roots, so it is enforced on the **response side**: the augmenter marks the `User` and injects per-field markers only when an owner-private field is selected (a plain `author{login}` is never marked), and `RedactDeniedOwnerPrivate` nulls those fields when the `user_private`/`gists` **category** is denied — by category, not the user's login, so a token holding only an `[[org]] name="<login>"` enumeration grant still cannot read the resolved user's private `email`/SECRET-gists/`savedReplies`/keys. `TestR35_UserPrivateFieldSetsCoupled` couples `gqlfilter.userPrivateFields` to the classifier's `viewerPrivateFieldCategory` so the front-gate and response sets cannot drift (round-35). The field's policy **category** is carried by the marker's PREFIX (set from the schema field NAME at augment time), not the aliasable response key, so a client alias (`mine: gists`) cannot downgrade `gists`→`user_private` (round-36). `pinnableItems`/`pinnedItems`/`itemShowcase` — whose `PinnableItem` (`Gist`|`Repository`) union surfaces the custodian's SECRET gists one structural hop past the coverage guard's single-`nodes` unwrap — are gated on `gists` too; `TestR31_ViewerPrivateFieldCoverage` now descends union/interface members so a future union-wrapped private member fails the build (round-36);
- `search(query:)` → one repo scope per `repo:` qualifier, else an unscoped `search`;
- `viewer` → unscoped `user` for identity, but each owner-private sub-resource (the custodian's own `gists`/`organizations`/`enterprises`/`projectsV2`/`savedReplies`/`email`/keys/…) emits its own denying `gists`/`user_private` category so a default-deny token can't read it (round-31/32); `rateLimit` and schema introspection (`__schema`/`__type`/`__typename`) → unscoped `meta`.

Variables are resolved. `operationName` is honored — only the executed operation is classified. The policy must allow **all** collected scopes. The AST walk is depth-bounded and fails closed on cyclic fragments / excessive nesting (an unbounded recursive walk would otherwise crash the process — `parser.ParseQuery` validates neither). The **response filter's `Augment`** has the same request-path CPU bounds: a token-limited pre-parse, then a **fragment-graph budget** (fragment-definition and total-spread-edge caps) applied *before* the schema validator's `Walk` — which validates every operation and fragment definition as a separate root and would otherwise re-traverse a densely cross-referencing fragment graph in ~O(N²) (tens of seconds for a few-hundred-KB document that sits under the token cap, billed before the policy verdict), so it fails closed in milliseconds (round-22) — plus the marker-injection and output-size caps that bound the rewrite itself.

### Node-ID requests — authoritative resolution

GraphQL requests can address objects by opaque node ID with no `repository()` scope — every mutation (`mergePullRequest(input:{pullRequestId:…})`) and `node(id:)`/`nodes(ids:)` reads. Guessing the repository from earlier reads is unsafe (a response for repo A can contain node IDs belonging to repo B via cross-references), so the proxy resolves authoritatively:

1. The classifier extracts **every** node-ID-shaped value from inline arguments and variables (id-typed keys; both modern `prefix_base64` and legacy base64 forms). It does **not** filter by type — there is no repo-scoped prefix allowlist to fall behind, so a repo-scoped object of any type is caught.
2. `proxy.resolveNodeScopes` looks each up in the verified `nodecache`; on a miss it asks GitHub a `nodes(ids:)` query **generated from the embedded schema** that requests `repository { nameWithOwner }` for every repo-scoped `Node` type (covering check runs, deployments, commits, … — not just the few that implement `RepositoryNode`, and including types whose only repo link is a union/interface, e.g. `RepositoryRuleset.source` → `... on Repository`), and caches the verified mapping plus the resolved `__typename` (30 min TTL). The cached `__typename` also fixes a mutation's per-resource key by the node's real type, not the mutation's name.
3. Each node that resolves to a **repository** becomes a scope the policy must allow at the request's access level. A node that resolves to an **org/user/enterprise-owned** Node type (Organization, Team, ProjectV2, audit entries, **Gist**, … — derived from the owner-private `@docsCategory` set) **fails closed**: it is not repo-attributable, so under `default=allow` an empty-scope `node(id:)` read of one would otherwise bypass an `[[org]]`/`gists` deny (the owner-level analogue of the repo-node fail-closed); read such an object via the policy-checked `organization(login:)`/`user(login:)`/`viewer` root instead. The GraphQL enterprise roots (`enterprise(slug:)`, plus the owner-private `enterpriseAdministratorInvitation`/`enterpriseMemberInvitation`, which carry the slug under `enterpriseSlug`) are each scoped to their slug as an org so an `[[org]]` rule gates them — a build-time guard (`TestQueryRootCoverage`) enumerates every `Query` root and fails unless it is classifier-scoped or in a justified public/token-gated allowlist, so a new owner-private root cannot silently fall to `default=allow`. A bare-`repositoryName` migration record (`RepositoryMigration`, reachable via `repository(){ owner{ …repositoryMigrations } }`) is **unconditionally redacted** by the response filter — it names a *different* repo than its enclosing marked ancestor, so ambient attribution would be unsound (round-21). A node that does **not** resolve (unknown/invalid, or not visible to the upstream token) adds no constraint — the upstream token cannot mutate what it cannot resolve, so this cannot exceed policy; and a request whose *only* nodes are unresolved carries no repo scope, which the policy denies as an unscoped write.

A node that resolves to a repo-scoped **type** but yields no repository (anomalous), a non-null node whose runtime `__typename` the **embedded schema does not recognize** (live schema drift — the resolve query never asked for its repository, so a repo-scoped drift type would otherwise slip through unmarked), a node whose runtime type is **repo-owned but has no derivable repository path** (by `@docsCategory` it belongs to a single repository, yet its only repo link is an argumented connection or a union whose members aren't `Repository` — e.g. `Workflow`, `DeployKey`, `ClosedEvent`, `DeploymentReview`, `RepositoryTopic`; such a node can be neither resolved nor response-tagged, so it must not be treated as a constraint-free non-repo node — round-16), an upstream error, or more than 100 IDs makes the request fail closed. Resolution is gated on `AllowsAnyWrite`/`AllowsAnyRead` so a token that can never act at that level does not trigger upstream calls. The `nodecache` only ever stores mappings the resolver verified — it is never populated by sniffing responses. (Resolving reads also closes a `mode = "allow"` gap: a `node(id:)` read of a blocked repo would otherwise fall through to the permissive default.)

## Policy engine

`policy.Evaluate(repo, org, access, resource, unscopedCategory)` resolves in order:

1. exact `[[repo]]` match (case-insensitive) → per-resource permission if present, else base `access`; unknown-resource writes under a permissioned rule fail closed;
2. exact `[[org]]` match (case-insensitive) → same;
3. `[defaults.unscoped][category]` when there is no repo/org;
4. unscoped writes with no repo/org are denied — unless step 3 already granted that category `read-write` (e.g. `[defaults.unscoped] gists = "read-write"` permits `POST /gists`);
5. `[defaults].mode` (`deny`/`allow`).

`Access` levels: `none` (nothing), `read` (read only), `read-write` (everything). For multi-scope requests the proxy ANDs the per-scope decisions. Per-resource **keys are matched case-insensitively**; a per-resource key on a `[[repo]]` rule is **validated** against the known resource set at mint/load (`policy.ValidateResourceKeys` / `classifier.KnownRepoResourceKeys`) so a typo (`contnets="none"`) is rejected, not silently degraded to base access. **Org** per-resource keys are open-ended (any org subpath segment), so they are **not** validated — a misspelled `[org.permissions]` key silently falls back to base org access (a fail-open footgun; verify the segment name, or deny at the base level). The REST org-direct per-resource gate (`/orgs/{org}/members`, `/teams`, `/projects`, `/rulesets`, …) has a GraphQL counterpart: the owner-root fields map to the matching `[org.permissions]` key — `membersWithRole`/`teams`/`mannequins`/`auditLog`/… → `members`/`teams`; `projectsV2`/`projects`/`recentProjects` → `projects`; `sponsorsActivities`/`monthlyEstimatedSponsorsIncomeInCents`/… → `sponsors`; `rulesets`/`ruleset` → `rulesets`; `repositoryCustomProperties` → `properties`; `interactionAbility` → `interaction-limits`; `ipAllowListEntries`/`domains`/`announcementBanner`/`organizationBillingEmail` → `settings` — so a `[org.permissions] members="none"`/`projects="none"`/`rulesets="none"`/… carve-out is enforced over GraphQL (on the `organization(login:)` root AND the `repository().owner` navigation path) exactly as on the REST sibling. The **Enterprise** root is gated identically by a parallel `gqlEnterpriseFieldToResource` map (`billingInfo`/`billingEmail` → `billing`, `organizations` → `organizations`, `securityContactEmail`/`ownerInfo`/`announcementBanner` → `settings`, `rulesets`/`repositoryCustomProperties` → `rulesets`/`properties`), since `enterprise(slug:)` otherwise degraded every field to base enterprise read and leaked the custodian enterprise's billing financials / security contact / private org inventory under a base `[[org]] name="<slug>"` read grant (round-37 org content → round-38 the rest + enterprise). A schema-derived guard (`TestR38_OwnerAdminContentCovered`) asserts every Organization/Enterprise field whose element `@docsCategory` is `orgs`/`enterprise-admin` is mapped — so a refreshed schema's new owner-admin field fails the build instead of degrading to base read. Because owner-private data on an `Organization`/`Enterprise` is reachable by MANY navigation paths (`organization(){teams{nodes{organization{membersWithRole}}}}`, `user(){organizations|enterprises{nodes{…}}}`, `repository().owner`, …) that a request-side scope cannot all enumerate, the durable enforcement is **response-side** (round-25/26): the augmenter injects the owner identifier (`login`/`slug`) onto every `Organization`/`Enterprise` object — plus a per-field marker carrying each member field's **response key** so a client *alias* (`roster: membersWithRole`) cannot dodge it — and `RedactDeniedOwnerPrivate` then, for each marked owner, (a) if the owner is BASE-denied (no org/enterprise access at all — e.g. `repository().owner` under a repo-only grant) keeps only public identity and nulls everything else (billing email, IP-allow-list CIDRs, verified domains, 2FA-enforcement, members, …) — *drift-proof*, it nulls everything not on a small public allowlist rather than enumerating the evolving private-field set — or (b) if only the `members` carve-out is denied, nulls exactly the member-identity fields by their marked response key. A `gqlfilter`/classifier test keeps the member field set in sync with the `members`-mapped request keys. The SAME response-side mechanism is generalized from the one hardcoded `members` resource to **every owner-private CONTENT resource** (rulesets/properties/settings/billing/projects/sponsors/organizations/packages/issue-types/…): the augmenter content-marks each such field with its per-resource key (carried alias-immune in the marker prefix, the field addressed by its alias-proof response key), and `RedactDeniedOwnerPrivate` nulls it when that resource is denied for the owner — so a `[org.permissions] rulesets="none"` carve-out is enforced on a base-ALLOWED org reached by ANY navigation path (`team{organization{rulesets}}`, `user(login:){organization(login:){rulesets}}`, deep `repository().owner`), not just the request-scoped roots the round-37/38 front gate covered (round-39). The enterprise member-org INVENTORY one hop below `enterprise(slug:){ownerInfo{…SettingOrganizations}}` is content-marked the same way and attributed to its ambient enterprise. `TestR39_OwnerContentResourceInSync` couples the response-side content map to the classifier's request maps so they cannot drift. A per-resource WRITE addressed by node id is likewise checked against BOTH the mutation-name key AND the node's real type-derived key (`unmarkIssueAsDuplicate` on a PullRequest → `issues`+`pulls`; `createLinkedBranch` on an Issue → `branches`+`issues`), requiring the policy to allow both so a permissive field name cannot ride past the node's stricter resource (round-25). The same union is applied to a write addressed by a STRING target (`createCommitOnBranch`'s `repositoryNameWithOwner` is a `CommittableBranch` → a branch tip, so it requires `contents`+`branches`), so the string form cannot bypass a `branches="none"` carve-out the equivalent `Ref`-node form is gated on (round-37).

## Token & secret model

- **Proxy tokens** (GHE mode): random 256-bit secrets, stored as SHA-256 hashes in `tokens.json` (`0600`), each carrying an embedded policy. Looked up in constant time; revocation and hard-deletion both go through the running server so they take effect immediately. `tokens.json` is written atomically (temp + rename).
- **Admin secret**: gates the `internal/web` admin API/UI (`POST /api/tokens` on `admin_bind`), a **second, independent** token-minting surface that issues a token with **any** policy on the admin secret **alone** — no GitHub-owner (TOFU) check, unlike the `/ui` sign-in path. Generated once and reused across restarts; file `0600`. Keep `admin_bind` on loopback and rotate the secret (`rm admin-secret`, restart) on host/owner compromise — deleting `owner.json` does **not** invalidate it (see README → operations).
- **Socket**: created with a restrictive umask so it is `0600` from the moment it exists; only the owning user can connect.
- **Real GitHub token (custodian)**: the proxy's single upstream credential, held only on the proxy host, never sent to clients. It is normally **captured by the first GitHub sign-in** (web `/ui` or `gh auth login`): the proxy runs GitHub's OAuth **device flow** (`internal/oauth`, the GitHub CLI's public app — no registration), resolves `viewer{login}`, and on the first sign-in records that login as the deployment **owner** and stores the captured token as the custodian (`internal/owner`, `owner.json`, `0600`) — trust-on-first-use. Every later sign-in must be that same owner and refreshes the captured token; a different login is denied. A pre-seeded token (`BGH_GITHUB_TOKEN` → `github_token` → the file written by `bgh-proxy login`) is an optional **fallback** custodian used only until a sign-in claims ownership. Storage is plaintext (encrypted-at-rest is a non-goal).

> **Claim before you expose (load-bearing).** Ownership is **first-sign-in-wins**: an exposed-but-unclaimed proxy can be permanently claimed by the first network-reachable stranger (the server starts and serves even when unclaimed — `main.go` only logs a warning). So the operator **MUST** claim the deployment over **loopback** — open `https://127.0.0.1:<port>/ui` and sign in, or pre-seed `BGH_GITHUB_TOKEN` (which binds the TOFU claim to the fallback custodian's own GitHub identity) — **before** binding/exposing the GHE listener. See [docs/deployment.md](docs/deployment.md) (Tailscale Serve / Caddy, `external_url`, loopback bind). Note: `gh auth login --hostname 127.0.0.1:<port>` does **not** work for the initial loopback claim once `external_url` points at the public name (the device-flow URL `gh` shows then points at the not-yet-exposed public host); use the `/ui` web route, which shows GitHub's own `github.com/login/device` URL.

## Security properties & threat model

The in-scope adversaries are: (1) a holder of a policy-restricted **proxy token** (GHE mode); (2) any **local process running as the user** (socket mode); (3) an **on-path network attacker** between client and proxy (mitigated by TLS and by stripping custodian-reach response headers); and (4) the **upstream GitHub response itself**, which can contain cross-repo data the response filters must redact (the filters are written to treat upstream data as adversarial). The intended invariant: **a client cannot exceed its policy**, even though the upstream token can. A load-bearing assumption is that **security does not depend on client behaviour** — the proxy enforces policy regardless of what `gh`/`curl`/any client does (the bundled `cli/` is the upstream gh submodule, out of scope).

**Where the boundary lives.** By default the custodian is the **broad** token captured from the operator's GitHub sign-in (`repo read:org gist workflow`) — avoiding fine-grained PATs is a goal of the project, not an oversight. Consequently **there is no GitHub-side floor in the default deployment: the proxy's classifier + policy + response filters *are* the boundary.** This is load-bearing, so both read paths fail closed: the GraphQL filter types each query against the embedded GraphQL schema and denies what it can't tag; the REST filter types each response against the embedded **OpenAPI description** (`internal/restfilter`), redacting repositories at the spec-derived locations and denying any path the spec doesn't describe. A fine-grained PAT pre-seeded as the custodian (`BGH_GITHUB_TOKEN` / `github_token`) is the *only* way to add a GitHub-enforced floor, and it deliberately re-introduces the PAT management the project avoids — so it is optional, not assumed below.

Enforced: deny-by-default; per-repo/org/resource read/write; multi-scope GraphQL (every touched repo checked); authoritative mutation scoping (no mis-attribution); case-insensitive name matching; path-traversal rejection; fail-closed on parse/resolve/complexity failures (the GraphQL parse is token-bounded, so a deeply nested query is denied, not crashed); resource bounds (10 MB request cap → 413; fail-closed filtered-response size cap; 100 k node-cache cap; 2048 live-grant cap → 503, device-flow rate limit → 429 — keyed on the request source address, so it is per-source on a direct listener but collapses to a single **global** cap behind a TLS-terminating front that reverse-proxies from loopback, *not* per-client isolation; 100-ID node-resolve cap); no token leakage to clients; constant-time secret comparison (including the grant device-/auth-code matchers); least-privilege file modes.

GraphQL **read isolation** is enforced by schema-aware **response filtering** (`internal/gqlfilter`), not by the classifier's scope check alone. The classifier still gates the entry point and unscoped categories, but the filter is what makes cross-repo navigation safe: the proxy types the read against GitHub's embedded schema, injects a hidden `repository { nameWithOwner }` tag **and a `__typename` tag** into every repo-scoped selection, forwards the rewritten query, and redacts from the JSON response every object the policy denies for its **(repository, resource)** — the resource derived from the tagged type (`PullRequest`→`pulls`, `Issue`→`issues`, …). Only the repository **container** (runtime type `Repository`) is kept leniently — via `Policy.CanReadAnything`, so a `base = "none"` + per-resource grant (e.g. "read only this repo's issues") doesn't null the container and lose its granted children. **Every other** repo-scoped object — including content types that map to no specific resource (`Discussion`/`Milestone`/`Project`/`Tag`/…, which fall to `metadata`) — must satisfy its `(repository, resource)` through the strict `Policy.Evaluate`, exactly as the direct path does; otherwise cross-repo navigation could forward such a content object's body under any non-`none` per-resource carve-out. For a selection written against an **interface/union** type (e.g. `... on Comment { body }`, or a field typed `ReferencedSubject`/`Node`), the marker is injected for every repo-scoped concrete type the selection could resolve to — derived from the schema, mirroring the `nodes(ids:)` resolve query — so the runtime object still self-identifies regardless of the abstract type it was selected through (without this, interface-typed selections went untagged and leaked). Because every repo-scoped datum self-identifies its real repository and type, this is sound against multi-root, `owner.repositories`, `forks`, `node(id:)`, search results, `viewer.repositories`, and interface/union selections alike, and a per-resource restriction (e.g. `pulls = "none"`) is enforced on navigated objects too, not just the entry point. A repo-owned content type with **no derivable repository path** (timeline events, `DeploymentReview`, `IssueFieldSingleSelectOption`, … — the same `@docsCategory`-derived set the node resolver fails closed on) is tagged with its `__typename` only and attributed to its nearest enclosing (marked) repository for the per-resource check, failing closed if it has no such ancestor — the navigation analogue of the `node(id:)` fail-closed, closing a per-resource read bypass where such an object reached by navigation under a marked parent of a *different* allowed resource was forwarded untagged (round-17). For a **cross-repository** such event (`CrossReferencedEvent`, attributed to its allowed ambient repo but whose `url`/`resourcePath` URI scalars name the *foreign* referencing issue/PR), those self-URL scalars are additionally nulled when they resolve to a repo policy denies — they carry no marker, so the `source`-content redaction alone left the denied repo's identity/existence visible (round-22; a schema-derived drift guard keeps the cross-repo-URI event set complete). The scrub is **value-driven** — it nulls every string field of the event whose value parses to a denied repo ref — so a client alias (`leak: url`) cannot move the scalar off its canonical key to dodge a key-name lookup (round-37, the same alias class as the owner-member and gist-category nulling). A `gqlfilter` build-time invariant test fails the build if the embedded schema gains a repo-bearing type the marker/resolve machinery does not cover, so a schema refresh can't silently reintroduce a fail-open gap. The **same filtering is applied to mutation response payloads** (a mutation's return selection is a read sub-graph), so a write grant on one repo cannot read a denied repo via what the mutation returns. A GraphQL request is **fully filtered or denied, never forwarded unfiltered**: if the filter is enabled but cannot type the request (a field newer than the embedded schema, an invalid query, or one that pre-declares the reserved marker alias), no response can be tagged/redacted, so the proxy **fails closed** (`respFilter == nil` → deny). It does not fall back to the classifier's `scanCrossRepoNav` denylist for typed-filter-eligible requests — that denylist is not complete enough to bound an untyped read, so relying on it could stream cross-repo data reached via an unlisted field. (`NavEscapes` is still used only when the filter is disabled entirely, e.g. in tests.)

For the filter to see plaintext, the proxy does **not** forward the client's `Accept-Encoding` upstream — Go's transport then negotiates compression itself and decompresses transparently, so every body can be typed and redacted (a gzipped body would otherwise be unparseable and forwarded unredacted). A GraphQL response the filter cannot parse is **denied**, not passed through (`filterGraphQLResponse` fails closed).

The REST enumeration/search endpoints that return repository-bearing entries from many repos (`/user/repos`, `/user/issues`, `/orgs/{org}/repos`, `/repos/{owner}/{repo}/forks`, `/issues`, `/notifications`, `/search/{repositories,code,issues,commits}`, the activity feeds `/{,orgs/{org}/,users/{u}/}events`, `/users/{u}/{starred,subscriptions}`, the org-wide alert feeds `/orgs/{org}/{secret-scanning,dependabot,code-scanning}/alerts`, `/orgs/{org}/teams/{team}/repos`, …) are filtered analogously by `internal/restfilter`: denied-repo entries are dropped from the array/`items` (keyed on `full_name` / `repository.full_name` / `repository_url` / the events `repo.name` / the `star+json` `repo.full_name`). This also closes a `[[org]]`-read + per-repo-`none` carve-out leak: org subpaths classify to the org only, so the alert feeds would otherwise expose a carved-out repo's alerts and cleartext secret-scanning secrets. Without this the `user`/`search`/`notifications` categories would let a client enumerate denied repos' metadata, read their code via `/search/code`, or read issue/PR titles via `/notifications`, over REST — bypassing the GraphQL filter. When matches are dropped from any repo-array (`items`, `repositories`, …), `total_count` is reduced by the dropped count (with `incomplete_results`) so it can't serve as a denied-repo existence/count oracle. **REST coverage and repo-locations are derived from GitHub's OpenAPI description (`internal/restfilter/gen` → `openapi_table.go`), not a hand-maintained allowlist** — so the whole REST surface is covered, and the filter fails closed like the GraphQL path: a recognized repo-bearing body that won't parse is denied, and a GET whose path is off-spec is denied unless the classifier already scoped it to one repository. A GET the table classifies as **`Pass`** (repo-free) is **not forwarded blind**: for a non-path-scoped (enumeration-shaped) response the proxy scans the **actual JSON body** for any repository identity the static table may have under-located (an untyped/opaque/cyclic response schema) and **fails closed if a denied repo is present** (`restfilter.ContainsDeniedRepo`, mirroring the generator's own `full_name`/`repository_url`/minimal-repo signals) — so `Pass` is sound even when the embedded spec under-describes a response. Binary downloads (non-JSON content types) and path-scoped responses stream untouched. The residual is spec freshness (a brand-new endpoint absent from the embedded spec is denied until regenerated — availability, not a leak). A handful of sibling shapes the spec-driven generator cannot locate are covered by small hand-maintained tables: bare-string repo arrays (`internal/restfilter/string_array_scrub.go`), an org-qualified bare `{id,name}` repo array (`org_named_repos.go`, e.g. `/orgs/{org}/attestations/repositories`), a nested bare-`name` `repositories[]` (`org_named_repos.go`, the org Copilot-metrics feeds), and Pass ops that name their repo only by an **opaque numeric id** the body-scan cannot map (`opaque_repo.go`, the Copilot `/agents/tasks[/{id}]` lookups, the `/orgs/{org}|/users/{u}/copilot-spaces*` feeds whose `resources_attributes[].metadata.repository_id` is numeric, webhook-delivery `repository_id`, and attestation-by-digest bundles) which **fail closed** when not path-scoped. Each hand table has a build-time completeness guard derived from the embedded spec (`TestSpecCoverage_{NestedBareNameRepos,OpaqueRepoIDOps}` — the latter flags any Pass GET op declaring a numeric `repository_id` with no mapping sibling), so a spec refresh that adds another such op fails the build instead of leaking; the broad opaque fail-closed is checked *after* the specific redactable tables so a wildcard opaque template (`/orgs/{org}/attestations/{subject_digest}`) does not shadow the literal redactable `/orgs/{org}/attestations/repositories`. The custodian's own private account data is also reachable third-person: the proxy authenticates *as* the custodian, so `/users/{username}/...` resolves to the custodian's account when `{username}` is their login. The first-person `/user/*` split (round-30) un-floors `/user/{emails,keys,…}` to the `user_private` category; the third-person `/users/{username}/*` is split the same way (round-35) — the authenticated-only subtrees (`settings/billing`, `projectsV2`, `docker`, `installation`, `copilot-spaces`) are `user_private` (denied unless explicitly granted, never via a bare `[[org]]` enumeration grant), while the public third-person feeds (`repos`/`keys`/`events`/…) stay org-scoped. The allowlist is **inverted** (public subtrees enumerated, everything else `user_private`) so a new authenticated-only subtree fails closed. The per-resource keep-gate is also applied to the cross-repo **content** feeds (`/user/issues`, `/issues`, `/search/issues`, `/search/code`, `/search/commits`, `/notifications`, and the **activity-event feeds** `/{,orgs/{org}/,users/{u}/}events` whose `payload.issue`/`pull_request` content is dropped when `issues` is denied — round-21) so a `base=read` + `<resource>=none` carve-out is enforced there too, and the cross-ref **scrub runs on write responses** (`POST`/`PATCH /pulls`, `POST`/`DELETE /pulls/{n}/requested_reviewers`, `PATCH /repos/{o}/{r}`, and the 202 fork-create endpoints `POST /forks` and `POST /repos/{o}/{r}/security-advisories/{ghsa}/forks` — a fork-originated PR's `head.repo` / a fork's `parent`/`source`/`template_repository`) not only GETs. The spec-coverage test now inspects 202 responses and checks **per cross-ref location** (not just presence of *some* scrub), so a partial scrub that covered `parent`/`source` but not `template_repository` is caught (round-22). The `content`-null scrub (the projectsV2 `.../items`/`.../drafts` linked Issue/PR content) descends an **array** root as well as an object — a `.../items` LIST returns a JSON array, which the object-only descent silently skipped while the op's presence in the content-scrub table suppressed the `Pass` body-scan backstop (round-36). The cross-ref scrub also **fails closed** when a cross-referenced issue carries only `repository_url` (the nested `repository` is schema-optional) rather than forwarding the foreign content. Separately, the proxy strips `X-OAuth-Scopes`/`X-Accepted-OAuth-Scopes`/`X-OAuth-Client-Id`, `X-GitHub-SSO`, `Github-Authentication-Token-Expiration`, and `Set-Cookie` from forwarded responses so the custodian token's scopes, SSO reach, and lifecycle are not disclosed to clients.

Explicitly **not** boundaries (see README → Security model): the response filter is only as current as its embedded schema (newer fields → fail closed); per-resource redaction is as complete as the type→resource map (a repo-scoped type not in the map falls back to repo-level/base access — these are wrappers/connections, timeline events gated by their parent, or no-resource-key types like discussions); **GraphQL connection/search counts leak** (`totalCount`/`issueCount`/… are computed by GitHub over the full set, so they reveal the count/existence of denied items even though their contents are redacted — `search(type:ISSUE){ issueCount }` is a content-existence oracle; not soundly closable in the filter because count fields can be aliased); the count oracle is an accepted residual (contents redacted, counts not), since the default deployment has no upstream floor to bound it — an optional fine-grained PAT custodian is the only thing that would, at the cost of the PAT management the project avoids; GraphQL `errors[]`/`extensions` (the parallel partial-failure channel GitHub fills beside `data`) are scrubbed too: any string naming a repository the policy cannot read is replaced with a generic message (round-25, fail-closed) — so a denied repo's name no longer rides out in a "no permission to view … in owner/repo" error beside the `data` the marker filter nulled; socket mode authenticates the user, not the process; node IDs are resolved against the embedded schema, so coverage of newly-added repo-scoped types tracks schema freshness (the `gqlfilter` coverage-invariant test turns a missed type into a build failure rather than a silent fail-open, and a repo-natured node that does not resolve fails closed). The **`/login/*` and `/ui` IdP endpoints are unauthenticated** (the token-acquisition surface); minting is gated by the GitHub owner sign-in (TOFU), not a proxy token, and the device-flow start endpoints are rate-limited and grant-capped — but the GHE listener should still be network-access-controlled (device-code phishing is inherent to the flow).

## Technology

- **Language**: Go (`net/http` server and client)
- **GraphQL parsing**: `github.com/vektah/gqlparser/v2`
- **Config/policy**: `github.com/BurntSushi/toml`
- **Crypto/TLS**: standard library (`crypto/*`, self-signed CA + leaf for GHE mode)

## Non-goals (not implemented)

mTLS team mode and per-identity client certs; multi-token upstream routing; encrypted token storage; glob patterns in rules; policy hot-reload; HA/clustering; an audit-query CLI. (Response-body filtering IS implemented and fail-closed for BOTH GraphQL — typed against the embedded GraphQL schema — and REST — typed against the embedded OpenAPI description, repo-locations derived mechanically per endpoint. Non-JSON / single-repo REST paths are scoped by the classifier; an off-spec path is denied. The remaining residual is spec freshness for both, not coverage.)

**API only, not git.** The proxy serves the GitHub REST + GraphQL APIs (`/api/v3`, `/api/graphql`) plus its own `/login`/`/ui`; it is not a git server. `gh repo clone` / `git push` through the proxy fail, and git transport is neither carried nor policy-filtered — policy governs API access only (file contents are reachable via the `contents` API, which IS filtered). Git access control to the underlying repos remains GitHub's job.
